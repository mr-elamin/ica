# Kubernetes Deep Dive: Command vs Args & Native Sidecars

This document clarifies how Kubernetes handles container execution and the modern "Native Sidecar" lifecycle.

---

## 1. `command` vs `args`

In Kubernetes, these fields override the settings in your Docker image.

| Field | Docker Equivalent | Description |
| :--- | :--- | :--- |
| **`command: []`** | `ENTRYPOINT` | The actual executable to run (e.g., `/bin/bash` or `python`). |
| **`args: []`** | `CMD` | The parameters passed to the executable (e.g., `-c`, `myscript.py`). |

### Rules of overriding:
1.  **If you supply only `args`:** They are passed to the `ENTRYPOINT` defined in the image.
2.  **If you supply only `command`:** It overrides the `ENTRYPOINT` and the image's `CMD` is ignored.
3.  **If you supply both:** The image's `ENTRYPOINT` and `CMD` are completely ignored. Your `command` runs with your `args`.

> **Note:** Both are represented as **Arrays (Strings)** in YAML, not Objects.
> ```yaml
> command: ["sh", "-c"]
> args: ["echo hello"]
> ```

---

## 2. The Native Sidecar Lifecycle (`restartPolicy: Always`)

Before K8s 1.28, sidecars were just "normal containers" that started *at the same time* as your app. This caused errors if the app tried to connect to the database before the proxy was ready.

### How `restartPolicy: Always` changed everything:
When an `initContainer` has `restartPolicy: Always`, it becomes a **Native Sidecar**.

### The Execution Flow:
1.  **Standard Init Phase:** `istio-init` runs. It must finish and **exit** (Exit Code 0).
2.  **Sidecar Init Phase:** `istio-proxy` starts. 
    *   Because it has `restartPolicy: Always`, Kubernetes **does not wait for it to exit**.
    *   Instead, Kubernetes waits for it to become **Ready** (via its Startup/Readiness probe).
3.  **Main Phase:** Once the sidecar is Ready, Kubernetes immediately starts the main application container (`httpbin`).

### Does it stop and start again?
**No.** 
*   It does **not** stop after the initialization period.
*   It does **not** restart when the main container starts.
*   It continues running the **exact same process** with the **exact same args** from start to finish.
*   The only difference is that now it "allows" the main container to start because it has reached a "Ready" state.

### Summary Table: Init vs Native Sidecar

| Feature | Standard Init (`istio-init`) | Native Sidecar (`istio-proxy`) |
| :--- | :--- | :--- |
| **Location** | `initContainers` | `initContainers` |
| **Restart Policy** | Default (None) | `Always` |
| **Success Criteria** | Must exit with 0. | Must pass Startup/Readiness probe. |
| **When App Starts** | After this container **dies**. | While this container **is running**. |
| **Termination** | Dead before app starts. | Dies only when the Pod is deleted. |
