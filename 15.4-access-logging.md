# 15.4 Istio Access Logging

Logging is the third pillar of mesh observability. While metrics give you numbers and tracing gives you the journey, **Access Logs** give you the "Black Box" recording of every single request.

---

## 1. The Anatomy of an Access Log
By default, Istio uses binary-level logging in Envoy. When you run `kubectl logs <pod-name> -c istio-proxy`, you see a long string of numbers and texts.

### Example Log Entry:
`[2024-01-25T23:00:43.001Z] "GET / HTTP/1.1" 200 - via_upstream - "-" 0 1683 3 2 "-" "curl/8.2.1" "0781918d-2349-9de7-b154-28e03d4c2224" "productpage:9080" "10.244.0.15:9080" inbound|9080|| 127.0.0.6:50011 10.244.0.15:9080 10.244.0.18:39774 outbound_.9080_._.productpage.default.svc.cluster.local default`

### The "Cheat Sheet" for Reading Logs:
*   `%START_TIME%`: When the request hit the proxy.
*   `%RESPONSE_CODE%`: `200`, `404`, `500`, etc.
*   `%RESPONSE_FLAGS%`: If this is not `-`, it tells you **why** a request failed (e.g., `NR` = No Route, `UH` = Upstream Unhealthy).
*   `%DURATION%`: How many milliseconds the whole thing took.
*   `%REQ(X-REQUEST-ID)%`: This is the **Trace ID**! You can copy this and search for it in Jaeger to see the trace.

## 1.1 Deep Dive: The Full Log Format Decoded

Every entry in the access log follows a specific template. Understanding this template is the difference between "I think it failed" and "I know exactly why it failed."

### The Template Breakdown
Let's deconstruct the standard Istio log format block by block:

| Field | Meaning | Real World Example |
| :--- | :--- | :--- |
| **`%START_TIME%`** | Timestamp the request was received. | `[2024-01-25T23:00:43.001Z]` |
| **`%REQ(:METHOD)%`** | HTTP Verb. | `GET` |
| **`%REQ(...PATH)%`** | The requested URL path. | `/api/v1/products` |
| **`%PROTOCOL%`** | The protocol used. | `HTTP/1.1` or `HTTP/2` |
| **`%RESPONSE_CODE%`** | Standard HTTP status code. | `200` |
| **`%RESPONSE_FLAGS%`** | **CRITICAL:** Envoy-specific error codes (see below). | `UH` (Unhealthy) or `DC` (Downstream Connection Termination) |
| **`%RESPONSE_CODE_DETAILS%`** | More info on the status code. | `via_upstream` |
| **`%BYTES_RECEIVED%`** | Body size of the request. | `0` (for a simple GET) |
| **`%BYTES_SENT%`** | Body size of the response. | `1683` |
| **`%DURATION%`** | Total time in milliseconds. | `150ms` |
| **`%REQ(USER-AGENT)%`** | What browser/tool made the call. | `curl/8.2.1` |
| **`%REQ(X-REQUEST-ID)%`** | The unique Correlation ID for tracing. | `0781918d-2349-9de7-b154-28e03d4c2224` |
| **`%UPSTREAM_HOST%`** | The IP:Port of the actual Pod that handled the request. | `10.244.0.15:9080` |
| **`%UPSTREAM_CLUSTER%`** | The Istio cluster name of the destination. | `outbound_.9080_._.productpage.default.svc.cluster.local` |
| **`%DOWNSTREAM_REMOTE_ADDRESS%`** | The IP of the **client** who called this proxy. | `10.244.0.18:39774` |
| **`%ROUTE_NAME%`** | The VirtualService route name. | `default` |

---

### Understanding `%RESPONSE_FLAGS%` (The Debugger's Gold)
If your `RESPONSE_CODE` is `503` or `404`, look at this field. It tells you who is really to blame:
*   **`-`**: No flag (Success).
*   **`UH`**: **Upstream Unhealthy.** All Pods in the target service are failing readiness checks.
*   **`UF`**: **Upstream Failure.** The connection was refused by the target Pod.
*   **`UO`**: **Upstream Overflow.** Circuit Breaker triggered (too many requests).
*   **`NR`**: **No Route.** Your VirtualService or DestinationRule is missing or wrong.
*   **`LH`**: **Local Health.** Envoy failed its own health check.
*   **`UT`**: **Upstream Timeout.** The app was too slow, and Envoy cut the connection.

## 2. Enabling Access Logs (The Telemetry Way)
Modern Istio (1.15+) prefers using the **Telemetry API** to enable logging, rather than global IstioOperator flags.

### Mesh-Wide Enablement:
```yaml
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: mesh-default
  namespace: istio-system
spec:
  accessLogging:
    - providers:
      - name: envoy # The built-in provider that sends logs to /dev/stdout
```

---

## 3. Power Feature: CEL Filtering
You don't want to log every single successful request in production (too much noise). You can use **CEL Expressions** to only log what matters.

### Example: Log ONLY Errors (5xx)
```yaml
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: error-logging-only
spec:
  accessLogging:
    - providers:
      - name: envoy  # The built-in provider that sends logs to /dev/stdout
      filter:
        expression: "response.code >= 500" # Only record if the app crashed
```

## 4. Customizing Log Formats (Extension Providers)
Sometimes the default log string is too hard to read. You can define your own format using **Extension Providers** in the `meshConfig`.

There are a couple of logging extension providers defined in the mesh config that we can use:
* Envoy file access log provider (envoyFileAccessLog)
* Envoy HTTP access logging service provider (envoyHttpAls)
* Envoy TCP access logging service provider (envoyTcpAls)
* Envoy Open Telemetry access logging service provider (envoyOtelAls)

### Step 1: Define the Provider (in Istio ConfigMap or Operator)
```yaml
meshConfig:
  extensionProviders:  # Envoy file access log provider
  - name: my-custom-json-log
    envoyFileAccessLog:
      path: /dev/stdout # Still send to standard output
      logFormat:
        json: # Use 'json' instead of 'labels' for structured logs
          status: "%RESPONSE_CODE%"
          method: "%REQ(:METHOD)%"
          path: "%REQ(:PATH)%"
          latency: "%DURATION%"
          trace_id: "%REQ(X-REQUEST-ID)%"
  - name: envoy-file-log
    envoyFileAccessLog:
      path: /var/log/mylog.log
      logFormat:
        text: "%RESPONSE_CODE%: path=%REQ(:PATH)% latency=%DURATION% trace=%REQ(X-REQUEST-ID)%"
```

### Step 2: Activate the Provider
If you want to apply different filters to different providers (e.g., log everything to JSON but only 404s to a file), you must **separate them into different list items** in the `accessLogging` section.

```yaml
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: filtered-logging
spec:
  accessLogging:
    # Item 1: JSON provider (Unfiltered - logs everything)
    - providers:
      - name: my-custom-json-log

    # Item 2: File provider (Filtered - only logs 404s)
    - providers:
      - name: envoy-file-log
      filter:
        expression: "response.code == 404"
```

## 5. Advanced: Access Logging Service (ALS)
Instead of writing to a file/standard output, Envoy can stream logs over gRPC to a central server using **ALS**.

Available Providers:
1.  **envoyHttpAls**: Stream HTTP metadata to a collector.
2.  **envoyTcpAls**: Stream TCP connection metadata.
3.  **envoyOtelAls**: Stream logs in OpenTelemetry format (The modern standard!).

### Why use ALS?
*   **Performance:** It moves the logging overhead off the sidecar and into a dedicated streaming service.
*   **Centralization:** No need to scrape thousands of pod logs; the logs arrive at the collector in real-time.

---

## 6. Summary: The 3 Ways to Log
| Method | Target | Use Case |
| :--- | :--- | :--- |
| **Default Envoy** | `/dev/stdout` | Quick debugging with `kubectl logs`. |
| **Envoy File** | `/var/log/...` | Legacy apps that expect logs on disk. |
| **ALS / OTel** | Remote Server | High-scale production environments with central logging (Splunk, Loki). |
