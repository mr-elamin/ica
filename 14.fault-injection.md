# Chapter 14 — Fault Injection (Chaos Engineering)

Fault Injection is a powerful "Chaos Engineering" tool. Instead of waiting for a real failure to occur, you **intentionally** break your traffic to see if your application handles it gracefully.

## 1. Why use Fault Injection?
*   **Test Timeouts:** Does your frontend lock up if the backend takes 10 seconds to respond?
*   **Test Error Handling:** Does your site crash if the "Recommendations" service returns a 500 error?
*   **Verify Fallbacks:** Does your app show "Cached Data" when the primary DB is "down"?

---

## 2. Configuration: Aborts vs Delays

Faults are configured in the **`VirtualService`**, not the DestinationRule, because they are part of the **routing logic**.

### A. HTTP Abort (Simulating Crashes)
This returns a specific error code to the caller immediately without even trying to hit the backend.

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: backend-test
spec:
  hosts:
  - backend.default.svc.cluster.local
  http:
  - fault:
      abort:
        httpStatus: 503     # The error code to return
        percentage:
          value: 10         # Only fail 10% of the time (randomly)
    route:
    - destination:
        host: backend.default.svc.cluster.local
```

### B. HTTP Delay (Simulating Latency)
This forces the request to "pause" inside the Envoy proxy before being sent to the backend.

```yaml
  http:
  - fault:
      delay:
        fixedDelay: 5s      # Pause for 5 seconds
        percentage:
          value: 100        # Delay EVERY request
    route:
    - destination:
        host: backend.default.svc.cluster.local
```

### C. gRPC Abort (Simulation for Microservices)
If your services communicate via gRPC, you can abort requests using specific gRPC status codes.

```yaml
  http:
  - fault:
      abort:
        grpcStatus: "PERMISSION_DENIED" # Note: gRPC status is a STRING
        percentage:
          value: 1
    route:
    - destination:
        host: backend.default.svc.cluster.local
```

> **Important Difference:**  
> *   `httpStatus` takes an **Integer** (e.g., `503`).  
> *   `grpcStatus` takes a **String** (e.g., `"PERMISSION_DENIED"`).  
> *   [Official list of gRPC status codes](https://github.com/grpc/grpc/blob/master/doc/statuscodes.md).

## 3. Advanced Strategy: Targeted Fault Injection
You usually don't want to break the site for real users. You can target faults to specific users (e.g., your own browser) using **headers**.

```yaml
  http:
  - match:
    - headers:
        x-user:
          exact: "qa-tester" # ONLY break the app for the QA team
    fault:
      abort:
        httpStatus: 500
        percentage:
          value: 100
    route:
    - destination:
        host: backend.default.svc.cluster.local
  - route:                   # Normal production traffic (no fault)
    - destination:
        host: backend.default.svc.cluster.local
```

---

## 4. Visual Summary

| Type | Impact on Caller | Simulates... |
| :--- | :--- | :--- |
| **Abort** | Instant error (5xx) | Server crash, DB connection refused, process killed. |
| **Delay** | Hanging request / Spinning wheel | Network congestion, high CPU load, disk I/O wait. |

### Does it actually hit the backend?
**No.** For **Aborts**, the Envoy proxy returns the 503 itself. For **Delays**, Envoy holds the request. This preserves your backend resources while you test your frontend's reaction.
