## 1. Does `istioctl install` install CRDs?

**YES.** When you run `istioctl install`, it installs Istio CRDs into the cluster as part of the process.

The CRDs are bundled inside the Istio CLI itself (under `manifests/` in the downloaded Istio package). The CLI applies them to the cluster when it runs. That is why after installation you can see these new resource types:

```bash
kubectl api-resources | grep istio
```

Output (after Istio is installed):
```
destinationrules      dr    networking.istio.io/v1       true   DestinationRule
envoyfilters               networking.istio.io/v1alpha3  true   EnvoyFilter
gateways              gw    networking.istio.io/v1       true   Gateway
virtualservices       vs    networking.istio.io/v1       true   VirtualService
serviceentries        se    networking.istio.io/v1       true   ServiceEntry
sidecars                   networking.istio.io/v1       true   Sidecar
authorizationpolicies ap    security.istio.io/v1         true   AuthorizationPolicy
peerauthentications   pa    security.istio.io/v1         true   PeerAuthentication
...
```

These are the CRDs that Istio installed so you can manage the service mesh using Kubernetes-native objects.

## 2. Why is `IstioOperator` not in `k api-resources`?

This is an **excellent and important question**.

### The Short Answer
`IstioOperator` is a **configuration schema consumed by the `istioctl` CLI** — it is NOT a Kubernetes CRD (in default installations). It is processed entirely **client-side** by `istioctl` before anything is applied to the cluster.

### The Long Answer

```
k api-resources | grep -i IstioOperator
# → returns nothing (unless you have the deprecated in-cluster Operator)
```

When you write a YAML file like this:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
metadata:
  name: control-plane
spec:
  profile: minimal
  meshConfig:
    trustDomain: acmellc.local
```

You are NOT applying this to the cluster directly. You pass it to `istioctl`:

```bash
istioctl install -f my-operator.yaml
```

`istioctl` reads this file, processes it, merges it with the selected profile, generates the actual Kubernetes manifests (Deployments, Services, ConfigMaps, etc.), and THEN applies them to the cluster.

### Timeline of IstioOperator:
| Phase | What happened |
|-------|--------------|
| Before v1.23 | There was an **in-cluster Operator** — a separate K8s Deployment that watched for `IstioOperator` CRDs and reconciled them. This made `IstioOperator` a real K8s resource. |
| v1.23 | The in-cluster Operator was **deprecated** |
| v1.24+ | The in-cluster Operator was **removed** |
| Today | `IstioOperator` is purely a **CLI configuration format**, not a K8s CRD |

### So what IS installed in the cluster?
```bash
# These are the real resources that end up in your cluster:
kubectl get deployment -n istio-system
kubectl get configmap -n istio-system
kubectl get service -n istio-system
kubectl get mutatingwebhookconfigurations | grep istio
```

The `IstioOperator` YAML is just the **input spec** you give to `istioctl`. Think of it like a Dockerfile — it's not running in Docker, it's the input to `docker build`.

## 4. Configuration Profiles - Deep Dive

Profiles are located in `manifests/profiles/` within the downloaded Istio package.

```bash
ls manifests/profiles/
# ambient.yaml  default.yaml  demo.yaml  empty.yaml
# minimal.yaml  openshift.yaml  preview.yaml  remote.yaml  stable.yaml
```

Each profile is itself an `IstioOperator` YAML that pre-configures which components are enabled/disabled.

### Profile Comparison Table

| Profile | istiod | IngressGW | EgressGW | ztunnel | Use Case |
|---------|--------|-----------|----------|---------|----------|
| `default` | ✅ | ✅ | ❌ | ❌ | **Production** |
| `demo` | ✅ | ✅ | ✅ | ❌ | Dev/showcase, high tracing |
| `minimal` | ✅ | ❌ | ❌ | ❌ | Minimal control plane only |
| `empty` | ❌ | ❌ | ❌ | ❌ | Custom base, enable only what you need |
| `ambient` | ✅ | ❌ | ❌ | ✅ | Ambient mode (sidecar-less) |
| `remote` | ❌ | ❌ | ❌ | ❌ | Multi-cluster remote cluster |
| `preview` | ✅ | ✅ | ❌ | ❌ | Experimental features |
| `openshift` | ✅ | ✅ | ❌ | ❌ | OpenShift + Istio CNI |

### What does the `default` profile look like internally?

```yaml
# manifests/profiles/default.yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  components:
    base:
      enabled: true        # installs CRDs
    pilot:
      enabled: true        # installs istiod
    ingressGateways:
    - name: istio-ingressgateway
      enabled: true        # installs ingress gateway
    egressGateways:
    - name: istio-egressgateway
      enabled: false       # NOT installed by default
```

### What does the `minimal` profile look like?

```yaml
# manifests/profiles/minimal.yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  components:
    base:
      enabled: true
    pilot:
      enabled: true
    ingressGateways:
    - name: istio-ingressgateway
      enabled: false       # No gateway — minimal!
```

### What does the `empty` profile look like?

```yaml
# manifests/profiles/empty.yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  components:
    base:
      enabled: false       # Nothing enabled at all
    pilot:
      enabled: false
```

`empty` is the **"blank canvas"** — you explicitly enable only what you need. This is great for GitOps where you want maximum control.

### Layering / Overlaying Profiles

You can layer profiles on top of each other:

```bash
# Start with minimal, overlay your custom config on top
istioctl install -f minimal-base.yaml -f my-customizations.yaml
```

Files are overlaid left to right — later files override earlier ones.

## 5. Option 1: Install with istioctl

### Basic installation:
```bash
istioctl install
# Uses 'default' profile if no profile specified
```

### Install with a specific profile:
```bash
istioctl install --set profile=demo
```

### Install with a custom IstioOperator file:
```bash
istioctl install -f my-operator.yaml
```

### Install with revision label (for canary upgrades):
```bash
istioctl install -y -f my-operator.yaml --revision prod
# This creates: istiod-prod (instead of istiod)
# Pods label for injection: istio.io/rev=prod
```

### Validate before applying:
```bash
istioctl install --dry-run -f my-operator.yaml
```

### Check what will be installed:
```bash
istioctl manifest generate -f my-operator.yaml | kubectl diff -f -
```

### Pros & Cons:
| Pros | Cons |
|------|------|
| Best validation and error reporting | Must manage multiple CLI versions for upgrades |
| Extensive customization | CLI flags can differ between environments |
| Health checks included | Not pure declarative / GitOps unfriendly |
| `--revision` enables canary upgrades | |

## 6. Option 2: Install with kubectl (manifest generate)

### Generate manifests then apply:
```bash
# Generate the manifest
istioctl manifest generate --set profile=minimal > minimal-profile.yaml

# Apply to cluster
kubectl apply -f minimal-profile.yaml --prune --all
```

### With a custom IstioOperator file:
```bash
istioctl manifest generate -f my-operator.yaml > generated.yaml
kubectl apply -f generated.yaml
```

### Compare with current cluster state:
```bash
istioctl manifest generate -f my-operator.yaml | kubectl diff -f -
```

### Pros & Cons:
| Pros | Cons |
|------|------|
| Pure Kubernetes manifests (GitOps friendly) | No configuration validation |
| Store in Git, apply with ArgoCD/Flux | Worse error reporting than `istioctl install` |
| No need to have `istioctl` in CD pipeline | `--prune --all` can accidentally delete resources |
| | Must re-run generate when upgrading |

## 7. Option 3: Install with Helm - Deep Dive

### Helm Chart Architecture

```
istio/base          → CRDs (must be first)
     ↓
istio/cni           → (optional) Istio CNI plugin
     ↓
istio/istiod        → Control plane (istiod)
     ↓
istio/gateway       → Ingress/Egress gateways (optional, repeatable)
istio/ztunnel       → (optional, ambient mode only)
```

### Add Istio Helm repo:
```bash
helm repo add istio https://istio-release.storage.googleapis.com/charts
helm repo update
```

### Chart 1: `istio/base` — CRDs

This chart ONLY installs Istio's Custom Resource Definitions (CRDs). Must be installed first.

```bash
helm install istio-base istio/base \
  --namespace istio-system \
  --create-namespace \
  --version 1.26.3
```

What it installs:
- All Istio CRDs (`DestinationRule`, `VirtualService`, `Gateway`, `AuthorizationPolicy`, etc.)
- Cluster-wide RBAC rules
- The `IstioOperator` CRD itself (for Helm-based workflows, this CRD IS registered)

```bash
# Verify CRDs are installed
kubectl get crds | grep istio.io
```

### Chart 2: `istio/cni` — Optional CNI Plugin

```bash
helm install istio-cni istio/cni \
  --namespace kube-system \
  --version 1.26.3
```

What it does:
- Installs a DaemonSet on every node
- Handles iptables rules for traffic redirection WITHOUT needing an init container (`istio-init`) in each pod
- Required for OpenShift (where init containers running as root are restricted)
- Required for ambient mode

```bash
# See CNI DaemonSet
kubectl get daemonset -n kube-system istio-cni-node
```

### Chart 3: `istio/istiod` — Control Plane

This is the main chart. It installs `istiod` — the single control plane binary that handles:
- **Pilot**: service discovery and xDS configuration push to proxies
- **Citadel**: certificate authority (CA) for mTLS
- **Galley**: configuration validation and distribution

```bash
# See all configurable values
helm show values istio/istiod > istiod-values.yaml

# Install with defaults
helm install istiod istio/istiod \
  --namespace istio-system \
  --version 1.26.3

# Install with custom values
helm install istiod istio/istiod \
  --namespace istio-system \
  --values my-istiod-values.yaml \
  --version 1.26.3
```

**Example custom `istiod-values.yaml`:**
```yaml
# istiod-values.yaml
pilot:
  replicaCount: 2
  resources:
    requests:
      cpu: 500m
      memory: 2Gi
    limits:
      cpu: 1000m
      memory: 4Gi

meshConfig:
  accessLogFile: /dev/stdout
  outboundTrafficPolicy:
    mode: REGISTRY_ONLY
  trustDomain: acmellc.local

global:
  proxy:
    resources:
      requests:
        cpu: 100m
        memory: 128Mi
      limits:
        cpu: 200m
        memory: 256Mi
```

### Chart 4: `istio/gateway` — Ingress/Egress Gateway

This chart can be installed **multiple times** to create multiple gateways.

```bash
# Install default ingress gateway
helm install istio-ingressgateway istio/gateway \
  --namespace istio-system \
  --version 1.26.3

# Install a second gateway in a different namespace
helm install app-gateway istio/gateway \
  --namespace my-app \
  --create-namespace \
  --set service.type=ClusterIP \
  --version 1.26.3

# Install an egress gateway
helm install istio-egressgateway istio/gateway \
  --namespace istio-system \
  --set service.type=ClusterIP \
  --version 1.26.3
```

**Example custom gateway values:**
```yaml
# gateway-values.yaml
replicaCount: 2

service:
  type: LoadBalancer
  ports:
  - name: http2
    port: 80

  - name: https
```

### Chart 5: `istio/ztunnel` — Ambient Mode Only

Only used when deploying Istio in **ambient mode** (no sidecars). Installs the ztunnel DaemonSet which handles Layer 4 traffic.

```bash
helm install ztunnel istio/ztunnel \
  --namespace istio-system \
  --version 1.26.3
```

### Full Helm Install Sequence (Production Example):
```bash
# 1. CRDs
helm install istio-base istio/base -n istio-system --create-namespace

# 2. Control plane
helm install istiod istio/istiod -n istio-system \
  --values istiod-values.yaml

# 3. Wait for istiod to be ready
kubectl rollout status deployment/istiod -n istio-system

# 4. Gateway
helm install istio-ingressgateway istio/gateway \
  -n istio-system \
  --values gateway-values.yaml
```

### Helm Pros & Cons:
| Pros | Cons |
|------|------|
| Standard Helm workflows (ArgoCD, Flux) | Values schema differs from IstioOperator API |
| Separate lifecycle for CRDs, control plane, gateways | Must know Helm values structure separately |
| GitOps native | No `istioctl` validation |
| Upgrade per-component independently | |

## 8. Can Istio replace Cilium as CNI?

### Short Answer: **NO — they are not the same thing and serve different purposes.**

### What each does:

| Feature | Cilium | Istio CNI |
|---------|--------|-----------|
| **Primary role** | Full CNI (pod networking, IP assignment, routing) | Only handles iptables interception for Istio proxies |
| **Layer** | L3/L4/L7 | L7 (service mesh) |
| **Pod IP assignment** | ✅ Yes | ❌ No |
| **Network policies** | ✅ Native K8s + CiliumNetworkPolicy | ✅ AuthorizationPolicy (app-level) |
| **eBPF** | ✅ Yes | ❌ No |
| **mTLS** | ❌ No (WireGuard optional) | ✅ Full mTLS |
| **Traffic management** | ❌ Limited | ✅ Full (routing, retries, circuit breaking) |
| **Observability** | ✅ Hubble | ✅ Kiali, Jaeger, Prometheus |

### Can they coexist?
**YES!** Cilium (as CNI) + Istio (as service mesh) is a **common production setup**:
- Cilium handles pod networking (IP assignment, routing, eBPF acceleration)
- Istio handles service mesh features (mTLS, traffic management, observability)
- Use `istio/cni` chart to avoid init containers — Istio CNI plugin works alongside Cilium

### Migration Considerations:
Replacing Cilium with Istio CNI alone is **NOT possible** — Istio CNI is not a full CNI. You would need to keep Cilium (or another CNI like Calico/Flannel) for basic pod networking.

```bash
# WRONG assumption — this won't work:
cilium uninstall
helm install istio-cni istio/cni ...   # ❌ Pods will lose networking

# CORRECT — Cilium handles networking, Istio CNI handles proxy interception:
helm install istio-cni istio/cni -n kube-system  # ✅ Runs alongside Cilium
```

## 9. IstioOperator API - Full Breakdown

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
metadata:
  name: my-operator
spec:
  # ── Top-level fields ──────────────────────────────
  profile: default          # Base profile to start from
  hub: docker.io/istio      # Docker registry for images
  tag: 1.26.3               # Istio version tag
  namespace: istio-system   # Namespace for control plane
  revision: prod            # Revision label (for canary upgrades)

  # ── Section 1: Global mesh config ────────────────
  meshConfig:
    trustDomain: cluster.local
    accessLogFile: /dev/stdout
    outboundTrafficPolicy:
      mode: ALLOW_ANY

  # ── Section 2: Component configuration ───────────
  components:
    base:
      enabled: true
    pilot:
      enabled: true
      k8s:
        replicaCount: 2
    ingressGateways:
    - name: istio-ingressgateway
      enabled: true
    egressGateways:
    - name: istio-egressgateway
      enabled: false

  # ── Section 3: Pass-through Helm values ──────────
  values:
    global:
      proxy:
        resources:
          requests:
            cpu: 100m
```

## 10. Section 1: meshConfig

`meshConfig` controls **global defaults** for the entire mesh. These settings apply to all proxies unless overridden per-workload.

### Key meshConfig Fields:

```yaml
meshConfig:

  # ── Trust & Identity ──────────────────────────
  trustDomain: acmellc.local
  # The SPIFFE trust domain. Workload SPIFFE IDs will be:
  # spiffe://acmellc.local/ns/<namespace>/sa/<serviceaccount>

  # ── Access Logging ────────────────────────────
  accessLogFile: /dev/stdout
  # Enable access logs on all proxies. Set to "" to disable.
  accessLogFormat: ""
  # JSON format example:
  # accessLogFormat: '{"method":"%REQ(:METHOD)%","path":"%REQ(X-ENVOY-ORIGINAL-PATH?:PATH)%"}'

  # ── Outbound Traffic Policy ───────────────────
  outboundTrafficPolicy:
    mode: REGISTRY_ONLY
    # ALLOW_ANY: allow traffic to any host (default)
    # REGISTRY_ONLY: only allow traffic to services in the mesh registry
    # REGISTRY_ONLY is more secure — blocks accidental external calls

  # ── Default Proxy Config ──────────────────────
  defaultConfig:
    # Applied to every sidecar proxy in the mesh
    meshId: my-mesh
    drainDuration: 45s
    connectTimeout: 10s
    proxyStatsMatcher:
      inclusionPrefixes:
      - "cluster"
      - "listener"
    tracing:
      zipkin:
        address: zipkin.istio-system:9411

  # ── Extension Providers (for telemetry) ───────
  extensionProviders:
  - name: jaeger
    opentelemetry:
      service: jaeger-collector.monitoring.svc.cluster.local
      port: 4317
  - name: prometheus
    prometheus: {}

  # ── mTLS Settings ──────────────────────────────
  meshMTLS:
    minProtocolVersion: TLSV1_2

  # ── Discovery Selectors ────────────────────────
  # Only watch namespaces with this label (reduces istiod load)
  discoverySelectors:
  - matchLabels:
      istio-discovery: enabled
```

### Override per-pod with annotation:
```yaml
# In your Deployment pod template:
annotations:
  proxy.istio.io/config: |
    tracing:
      zipkin:
        address: zipkin.monitoring:9411
    drainDuration: 60s
```

## 11. Section 2: Components

### 11.1 BaseComponentSpec (`base`)

The `base` component is responsible for installing Istio's **CRDs and cluster-wide RBAC**.

```yaml
components:
  base:
    enabled: true   # Default: true in all profiles except 'empty'
    # If disabled, no CRDs are installed.
    # Disable ONLY when CRDs are already installed (e.g., by Helm base chart)
    # or when setting up a remote cluster that doesn't manage CRDs.
```

**When would you disable it?**
- In the `remote` profile — the remote cluster doesn't install CRDs, the central cluster does
- When using the `empty` profile as a base and managing CRDs separately

### 11.2 ComponentSpec (`pilot`, `cni`, `ztunnel`, `istiodRemote`)

`ComponentSpec` is used for single-instance components. Fields available:

```yaml
components:
  pilot:                          # = istiod
    enabled: true
    namespace: istio-system       # where to deploy
    hub: docker.io/istio          # override image hub
    tag: "1.26.3"                 # override image tag
    k8s:
      replicaCount: 3
      hpaSpec:
        minReplicas: 2
        maxReplicas: 5
        metrics:
        - type: Resource
          resource:
            name: cpu
            target:
              type: Utilization
              averageUtilization: 80
      resources:
        requests:
          cpu: 500m
          memory: 2Gi
        limits:
          cpu: 1000m
          memory: 4Gi
      nodeSelector:
        node-role: control-plane
      tolerations:
      - key: dedicated
        operator: Equal
        value: istio
        effect: NoSchedule
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchLabels:
                  app: istiod
              topologyKey: kubernetes.io/hostname
      env:
      - name: PILOT_ENABLE_STATUS
        value: "true"
      - name: PILOT_TRACE_SAMPLING
        value: "100"
      podAnnotations:
        prometheus.io/scrape: "true"
        prometheus.io/port: "15014"
      serviceAnnotations:
        service.beta.kubernetes.io/aws-load-balancer-type: nlb
      strategy:
        rollingUpdate:
          maxSurge: 1
          maxUnavailable: 0
      podDisruptionBudget:
        minAvailable: 1

  cni:
    enabled: false  # Enable if you want Istio CNI
    namespace: kube-system

  ztunnel:
    enabled: false  # Enable for ambient mode

  istiodRemote:
    enabled: false  # Enable for remote cluster in multi-cluster setup
    injectionPath: /inject
```

### 11.3 GatewaySpec (`ingressGateways`, `egressGateways`)

`GatewaySpec` is an array (you can have multiple gateways). It extends `ComponentSpec` with:
- `name`: the gateway deployment name
- `label`: labels applied to the gateway Kubernetes Service and Pod

```yaml
components:
  ingressGateways:
  - name: istio-ingressgateway     # default gateway from 'default' profile
    enabled: true
    namespace: istio-system
    label:
      istio: ingressgateway
    k8s:
      replicaCount: 2
      service:
        type: LoadBalancer
        ports:
        - name: status-port
          port: 15021
          targetPort: 15021
        - name: http2
          port: 80
          targetPort: 8080
        - name: https
          port: 443
          targetPort: 8443
        - name: tls
          port: 15443
          targetPort: 15443
      resources:
        requests:
          cpu: 100m
          memory: 128Mi
        limits:
          cpu: 2000m
          memory: 1Gi
      hpaSpec:
        minReplicas: 2
        maxReplicas: 10

  - name: internal-gateway          # second gateway in different namespace
    enabled: true
    namespace: internal-apps
    label:
      gatewayType: internal
    k8s:
      service:
        type: ClusterIP              # Internal only — no LoadBalancer
        ports:
        - name: http2
          port: 80
          targetPort: 8080
        - name: https
          port: 443
          targetPort: 8443
        - name: status-port
          port: 15021
          targetPort: 15021
        - name: custom-app-port
          port: 5000
          targetPort: 5000

  egressGateways:
  - name: istio-egressgateway
    enabled: true
    namespace: istio-system
    label:
      istio: egressgateway
    k8s:
      service:
        type: ClusterIP              # Egress gateways don't need external IPs
      replicaCount: 2
```

## 12. Ingress & Egress Gateway — Logic and Purpose

### 12.1 Infrastructure vs. Configuration
One of the most common sources of confusion is distinguishing between the **Proxy pod** and the **Config resource**.

| Concept | What it is | Resource Type | Created By |
|---------|------------|---------------|------------|
| **The Infrastructure** | The actual Envoy proxy pod + K8s Service. | `Deployment`, `Service`, `HPA` | `istioctl install` (IstioOperator) |
| **The Listener** | Defines which ports and hostnames the proxy listens to. | `Gateway` (istio.io) | You (Manual YAML) |
| **The Router** | Defines where traffic goes *after* it hits the proxy. | `VirtualService` | You (Manual YAML) |

### 12.2 Does it create a LoadBalancer?
*   **Ingress Gateway:** **YES.** By default, the `istio-ingressgateway` component in the `IstioOperator` creates a Kubernetes `Service` of `type: LoadBalancer`.
*   **Egress Gateway:** **NO (Usually).** By default, it creates a `ClusterIP`.

### 12.3 Routing Traffic (The Virtual Host Logic)
To route traffic to a specific app, you **must** use a `VirtualService`.
1.  **Traffic Source:** External client hits the LoadBalancer IP.
2.  **Entry Point:** Ingress Gateway proxy receives it.
3.  **Selection:** The `Gateway` resource (CRD) identifies that "myapp.com" is allowed on port 443.
4.  **Routing:** The `VirtualService` sees the host "myapp.com" and says "Send this to the `prod-service` K8s service".

### 12.4 Istio Gateways vs. Kubernetes API Gateway
Istio now supports **two** ways to configure entry points. They **coexist**.

| Feature | Istio Classic API | Kubernetes Gateway API |
|---------|--------------------|------------------------|
| **CRDs Used** | `Gateway` (istio.io), `VirtualService` | `Gateway` (k8s.io), `HTTPRoute` |
| **Status** | Mature, standard for years. | The new industry standard (v1.0+). |
| **Auto-Prov** | Must define gateways in `IstioOperator`. | Creating a `Gateway` CRD can auto-provision the proxy! |

### 12.5 What is an Istio Gateway?

An Istio Gateway is a **standalone Envoy proxy** deployed at the edge of your mesh to manage inbound/outbound traffic. It is a **separate Deployment** — NOT the sidecars running next to your apps.

```
Internet
   │
   ▼
[LoadBalancer Service]
   │
   ▼
[istio-ingressgateway Pod]   ← standalone Envoy proxy
   │   (Istio Gateway + VirtualService rules applied)
   ▼
[App Pod with sidecar]
   │
   ▼
[App Container]
```

### 12.6 Ingress Gateway Examples

```yaml
# Step 1: Gateway resource — defines what traffic to accept
apiVersion: networking.istio.io/v1
kind: Gateway
metadata:
  name: my-gateway
  namespace: istio-system
spec:
  selector:
    istio: ingressgateway   # targets the gateway Pod with this label
  servers:
  - port:
      number: 80
      name: http
      protocol: HTTP
    hosts:
    - "myapp.example.com"
  - port:
      number: 443
      name: https
      protocol: HTTPS
    tls:
      mode: SIMPLE
      credentialName: myapp-tls-cert   # K8s secret with TLS cert
    hosts:
    - "myapp.example.com"
---
# Step 2: VirtualService — routes traffic to your app
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: myapp-vs
spec:
  hosts:
  - "myapp.example.com"
  gateways:
  - istio-system/my-gateway
  http:
  - match:
    - uri:
        prefix: /api/v2
    route:
    - destination:
        host: myapp-v2.default.svc.cluster.local
        port:
          number: 8080
      weight: 90
    - destination:
        host: myapp-v1.default.svc.cluster.local
        port:
          number: 8080
      weight: 10
```

### Egress Gateway

**Purpose:** Control and monitor traffic LEAVING the cluster to external services.

**Why use it?**
- Centralize all outbound traffic through a single point
- Apply policies to external traffic (rate limiting, authentication)
- Audit/log all outbound connections
- Required when `outboundTrafficPolicy: REGISTRY_ONLY` is set — external services must be routed through the egress gateway

**How it works:**
1. Deploy `istio-egressgateway` (ClusterIP — no external IP needed)
2. Create a `ServiceEntry` to register the external service
3. Create a `Gateway` targeting the egress gateway pod
4. Create a `VirtualService` to route traffic from app → egress gw → external service

```yaml
# Register the external service
apiVersion: networking.istio.io/v1
kind: ServiceEntry
metadata:
  name: external-api
spec:
  hosts:
  - api.external-service.com
  ports:
  - number: 443
    name: https
    protocol: HTTPS
  resolution: DNS
  location: MESH_EXTERNAL
---
# Bind egress gateway to this external host
apiVersion: networking.istio.io/v1
kind: Gateway
metadata:
  name: egress-gateway
spec:
  selector:
    istio: egressgateway
  servers:
  - port:
      number: 443
      name: tls
      protocol: TLS
    tls:
      mode: PASSTHROUGH
    hosts:
    - api.external-service.com
---
# Route: app → egress-gw → external service
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: external-api-route
spec:
  hosts:
  - api.external-service.com
  gateways:
  - mesh                           # applies inside the mesh
  - egress-gateway                 # applies at the egress gateway
  tls:
  - match:
    - gateways:
      - mesh
      port: 443
      sniHosts:
      - api.external-service.com
    route:
    - destination:
        host: istio-egressgateway.istio-system.svc.cluster.local
        port:
          number: 443
  - match:
    - gateways:
      - egress-gateway
      port: 443
      sniHosts:
      - api.external-service.com
    route:
    - destination:
        host: api.external-service.com
        port:
          number: 443
```

## 13. Does installing a component install a new istiod?

**NO.** Installing an additional component (like a new gateway) does **NOT** install a new istiod.

Here is the breakdown:

| What you install | New istiod? | What actually deploys |
|-----------------|-------------|----------------------|
| `pilot` component | ✅ YES — this IS istiod | istiod Deployment |
| `ingressGateways` | ❌ NO | Gateway Deployment (Envoy) + Service |
| `egressGateways` | ❌ NO | Gateway Deployment (Envoy) + Service |
| `cni` | ❌ NO | CNI DaemonSet |
| `ztunnel` | ❌ NO | ztunnel DaemonSet |
| `base` | ❌ NO | CRDs only |

### The istiod controls ALL gateways:
```
istiod (pilot)
  │  xDS (Envoy config push)
  ├──→ ingress-gateway pod (Envoy)
  ├──→ egress-gateway pod (Envoy)
  ├──→ app-pod sidecar (Envoy)
  └──→ app-pod2 sidecar (Envoy)
```

All Envoy proxies (sidecars AND gateways) connect to the SAME `istiod` for their configuration. The gateway is just another Envoy — it doesn't run istiod inside it.

### What about `--revision`?

When you use `--revision prod`, istiod is named `istiod-prod`. Any gateway installed with `--revision prod` will also connect to `istiod-prod`:

```bash
# Install control plane
istioctl install -f control-plane.yaml --revision prod
# → creates: istiod-prod

# Install gateway (separate IstioOperator file)
istioctl install -f gateway.yaml --revision prod
# → creates: global-ingress gateway
# → gateway connects to: istiod-prod (same revision)
```

This is the **recommended pattern for production** — separate control plane and gateway installation, same revision.

## 14. Section 3: values (pass-through)

The `values` field passes configuration directly to the underlying Helm templates. Use this for settings that don't have a first-class (top-level not available in `meshConfig` or `components`) IstioOperator equivalent.

### Does istioctl use Helm?
**YES.** Under the hood, `istioctl` uses the exact same Helm charts that you would use if you ran `helm install istio/istiod`. 
- When you run `istioctl install -f operator.yaml`, the command looks into the `operator.yaml`, takes the settings, and "injects" them into the Helm templates before applying them to the cluster.

### Precedence: Who wins?
If you define the same setting in two places, Istio follows this order of priority:
1.  **Top-Level API (`meshConfig`, `components`, `hub`, `tag`)** ← **WINNER**
2.  **`values` field** ← Only used if NOT defined in top-level.

**Example:**
```yaml
spec:
  # This is the "First Class" API
  meshConfig:
    accessLogFile: "/dev/stdout"  # <--- THIS WINS

  # This is the "Pass-through" to Helm
  values:
    global:
      proxy:
        accessLogFile: "/tmp/logs" # <--- THIS IS IGNORED
```

### Mapping to `helm --set`:
The `values` section is written exactly like a Helm `values.yaml` file.
- `spec.values.global.proxy.image` in the YAML 
- is the same as:
- `helm install --set global.proxy.image=...`

```yaml
spec:
  values:
    global:
      # Image pull settings
      imagePullPolicy: IfNotPresent
      imagePullSecrets:
      - name: my-registry-secret

      # Proxy (sidecar) defaults
      proxy:
        image: proxyv2
        clusterDomain: cluster.local
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 200m
            memory: 256Mi
        logLevel: warning
        componentLogLevel: "misc:error"
        privileged: false
        enableCoreDump: false

      # Logging
      logging:
        level: "default:info"

      # Network
      network: ""              # for multi-cluster
      multiCluster:
        enabled: false
        clusterName: ""

    # istiod-specific values
    pilot:
      autoscaleEnabled: true
      autoscaleMin: 1
      autoscaleMax: 5
      replicaCount: 1
      traceSampling: 1.0
      env:
        PILOT_ENABLE_WORKLOAD_ENTRY_AUTOREGISTRATION: true

    # CNI values (when using istio/cni)
    cni:
      cniBinDir: /opt/cni/bin
      cniConfDir: /etc/cni/net.d
      excludeNamespaces:
      - istio-system
      - kube-system
```

**Rule of thumb:** If a setting exists in `meshConfig` or `components`, use those. Use `values` only for settings without a direct IstioOperator equivalent (e.g., image pull secrets, CNI paths).

## 15. Exam Tips: Common "Gotchas"

### 15.1 Component vs istiod
**Question:** If I add a second `ingressGateway` in a new `IstioOperator` file, does it deploy a second control plane?
**Answer:** **No.** It only deploys the Envoy proxy and a Service. It connects back to the existing `istiod`.

### 15.2 Preferred Configuration
**Question:** Is it preferred to use `values` or `meshConfig`?
**Answer:** `meshConfig`. The high-level API is validated by `istioctl` and is more stable. Use `values` only as a last resort.

### 15.3 Revision Labeling
**Question:** How does a namespace know which Istio version to use?
**Answer:** The label on the namespace. 
- `istio-injection=enabled` (Legacy/Default)
- `istio.io/rev=prod` (Modern/Revision-based)
