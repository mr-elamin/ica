# Istio PKI: Understanding the Chain of Trust

In Istio, security is built on **Mutual TLS (mTLS)**. To make this work, every workload needs a certificate. But who verifies these certificates? That is where the **PKI (Public Key Infrastructure)** and the **Chain of Trust** come in.

---

## 1. The Real-Life Analogy: The Passport System

To understand why we use Intermediate CAs instead of just one Root CA, think of a country's Passport system:

1.  **The Root CA (The Ministry of Foreign Affairs / Central Government):**
    *   They are the ultimate authority. 
    *   They have a "Master Stamp" (Private Key) that everyone in the country trusts.
    *   **The Air Gap Strategy:** The Root CA private key is kept on a computer that has **never touched the internet**. It is locked in a physical vault. If someone stole this, they could sign fake certificates for anyone on Earth (Google, Bank of America, etc.).
    *   They only wake it up once every few years to sign a new **Intermediate CA**.

2.  **The Intermediate CA (The Regional Passport Office / Consulate):**
    *   This is the "Working Office." They keep their keys on a server connected to the network.
    *   **Role:** They do the heavy lifting of signing thousands or millions of user certificates.
    *   **Security Benefit:** If an Intermediate CA is compromised, the damage is contained. The Root CA can stay in its vault and simply "Revoke" that specific intermediate.

3.  **The Entity Certificate (Your Personal Passport):**
    *   This is your website or Istio workload certificate. It expires quickly (usually 90 days for Let's Encrypt).

## 2. Advanced: How Revocation Works (The Kill Switch)

If an Intermediate CA key is stolen, how does your computer know?

### 2.1 CRL (Certificate Revocation List)
*   **What it is:** A big blacklist of serial numbers.
*   **The Problem:** The list gets massive over time. Your browser has to download the whole file every time it wants to check. It's too slow.

### 2.2 OCSP (Online Certificate Status Protocol)
*   **What it is:** Your computer asks the CA directly: *"Hey, is Cert #1234 still good?"*
*   **The Problem:** It leaks your privacy (the CA knows which websites you are visiting) and adds delay.

### 2.3 OCSP Stapling (The Modern Way)
*   **What it is:** The **Server** asks the CA for a "Signed Proof of Life" every 24 hours. When you visit the site, the server hands you the cert **and** the signed proof. 
*   **Benefit:** Your computer doesn't have to talk to the CA, and it's fast.

## 3. Hands-on Tutorial: Building a 3-Tier Chain with OpenSSL

We will create a **Root CA**, an **Intermediate CA**, and a **Second Intermediate CA**.

### 2.1 Setup the Workspace
```bash
mkdir -p ~/istio/pki-lab && cd ~/istio/pki-lab
```

### 2.2 Phase 1: The Root CA (The "God" Certificate)
The Root CA is **self-signed**. Its Issuer and Subject are the same.

```bash
# 1. Generate the Root Key
openssl genrsa -out root-key.pem 4096

# 2. Create the Root Certificate (Self-Signed)
openssl req -x509 -new -nodes -key root-key.pem -sha256 -days 3650 \
  -subj "/O=Acme LLC/CN=Acme Root CA" \
  -out root-cert.pem
```

### 2.3 Phase 2: First Intermediate CA
This one is signed by the **Root**.

```bash
# 1. Generate Key
openssl genrsa -out inter1-key.pem 3072

# 2. Create CSR (Application form)
openssl req -new -key inter1-key.pem \
  -subj "/O=Acme LLC/CN=Acme Intermediate CA 1" \
  -out inter1.csr

# 3. Sign the CSR with Root Key
openssl x509 -req -in inter1.csr \
  -CA root-cert.pem -CAkey root-key.pem -CAcreateserial \
  -out inter1-cert.pem -days 1000 -sha256
```

### 2.4 Phase 3: Second Intermediate CA
This one is signed by **Intermediate 1**.

```bash
# 1. Generate Key
openssl genrsa -out inter2-key.pem 3072

# 2. Create CSR
openssl req -new -key inter2-key.pem \
  -subj "/O=Acme LLC/CN=Acme Intermediate CA 2" \
  -out inter2.csr

# 3. Sign the CSR with Intermediate 1
openssl x509 -req -in inter2.csr \
  -CA inter1-cert.pem -CAkey inter1-key.pem -CAcreateserial \
  -out inter2-cert.pem -days 500 -sha256
```

## 3. The Logic of Inspecting the Chain

To verify a chain, you must see that each certificate's **Issuer** matches the **Subject** of the one above it.

### 3.1 Inspecting Subjects and Issuers
Run these commands to see the "linkage":

```bash
# Root: Issuer and Subject are the same
openssl x509 -in root-cert.pem -noout -subject -issuer

# Inter 1: Issued by Root
openssl x509 -in inter1-cert.pem -noout -subject -issuer

# Inter 2: Issued by Inter 1
openssl x509 -in inter2-cert.pem -noout -subject -issuer
```

### 3.2 Verifying the Chain mathematically
You can use `openssl verify` to check if the path is valid. To check `inter2`, you need to provide the "bridge" (`inter1`) and the "trust anchor" (`root`).

```bash
# Create a bundle of the chain (excluding the leaf)
cat inter1-cert.pem root-cert.pem > chain-bundle.pem

# Verify inter2 against the bundle
openssl verify -CAfile chain-bundle.pem inter2-cert.pem
# Output should be: inter2-cert.pem: OK
```

# 4. Practical PKI Facts (Real World)

### 4.1 Do I need to install Intermediates on my PC?
**No.** You only install the **Root CA** (the Trust Anchor). 
When you visit a website (e.g., `google.com`), the web server sends you **two** things:
1.  Its own certificate.
2.  The Intermediate CA certificate.

Your PC looks at the Intermediate, sees it was signed by a Root you already trust, and then uses that Intermediate to verify the website. This keeps your local "Trust Store" small and clean.

### 4.2 If an Intermediate CA is stolen, how do I find out?
Revocation is the hardest problem in security. There are 3 ways your PC learns about it:

1.  **Browser/OS Updates:** If a major CA (like Let's Encrypt) has an emergency, Apple/Google/Microsoft will release a software update within hours to **remove** that certificate from your device.
2.  **CRL/OCSP:** As discussed in Section 2, your computer can query the CA's servers.
3.  **One-CRL / CRLite:** Modern browsers (Chrome/Firefox) download a compressed version of "The world's bad certificates" directly into the browser every day.

### 4.3 The Air Gap
In the real world, the Root CA's private key is stored on a **HSM (Hardware Security Module)** inside a safe. The key can never be exported or seen by humans. To sign a new Intermediate CA, several authorized people must physically meet in a room, provide their physical keys, and perform a "Signing Ceremony."

## 5. The "cacerts" Convention in Istio

In the Istio documentation, you noticed we never told Istio to use our secret:
`kubectl create secret generic cacerts -n istio-system ...`

### Why it works:
Istio uses **Convention over Configuration**. 
- The Pilot (`istiod`) is programmed to look for a secret named **exactly** `cacerts`.
- If found, it mounts it to `/var/run/secrets/cacerts`.
- If it sees `ca-key.pem` in that folder, it switches from "Self-Signed Mode" to **"Plug-in CA Mode"**.

### What istiod expects in the secret:
The secret **must** contain these exact filenames as keys:
*   `root-cert.pem`: The ultimate trust anchor (Root CA certificate).
*   `ca-cert.pem`: Your intermediate certificate.
*   `ca-key.pem`: The private key used by Istio to sign workload certificates.
*   `cert-chain.pem`: The full path from your intermediate back to the root (Intermediate + Root).

**Warning:** If you create this secret *after* Istio is already running, you must restart `istiod` for it to pick up the new chain:
```bash
kubectl rollout restart deployment istiod -n istio-system
```

### Pro Tip: Viewing multiple certs in one file
Standard `openssl x509` only shows the 1st cert. Use this to see **EVERYTHING** in a bundle (like `cert-chain.pem`):
```bash
openssl crl2pkcs7 -nocrl -certfile cert-chain.pem | openssl pkcs7 -print_certs -text -noout
```
