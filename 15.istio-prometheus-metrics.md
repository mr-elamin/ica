# Chapter 15 — Istio & Envoy Prometheus Metrics Deep Dive

When you run `curl localhost:15020/stats/prometheus`, you are seeing the "Internal Diary" of the Envoy proxy. This guide summarizes the most critical metrics you need as an Istio Operator or Architect.

### Two Tiers of Metrics
In a Service Mesh, you primarily work with two types of metrics:
1.  **Proxy-Level Metrics (`envoy_` prefix)**: Low-level data about connections, circuit breaking, and proxy health.
2.  **Service-Level Metrics (`istio_` prefix)**: High-level data about requests, latency, and service-to-service communication.

---

## 1. Proxy-Level Metrics (`envoy_` prefix)
These tell you about the health and performance of the Envoy process itself.

| Metric | Example Value | Meaning |
| :--- | :--- | :--- |
| `envoy_server_uptime` | `3600` | The sidecar has been running for 1 hour (3600 seconds). |
| `envoy_server_memory_allocated` | `45000000` | Envoy is currently using ~45MB of RAM. |
| `envoy_server_version` | `1.23.1` | The current version of the Envoy binary. |
| `envoy_server_live` | `1` | `1` means the proxy is healthy and ready to process traffic. |

---

## 2. Upstream "Cluster" Metrics
These tell you about the **Services you are talking to** (The destinations).

| Metric | Example Value | Meaning |
| :--- | :--- | :--- |
| `envoy_cluster_upstream_rq_total` | `1500` | You have sent 1,500 requests to a specific backend (labeled by `cluster_name`). |
| `envoy_cluster_upstream_cx_active` | `2` | You have 2 persistent TCP connections open to the backend pods right now. |
| `envoy_cluster_membership_healthy` | `3` | Envoy knows about 3 healthy backend pods for this service. |
| `envoy_cluster_upstream_rq_timeout` | `5` | 5 requests failed because the backend took too long to answer. |

### Circuit Breaker & Resiliency Examples:
*   `envoy_cluster_upstream_rq_pending_overflow`: `10`
    *   **Meaning:** 10 requests were rejected because your "Waiting Room" (Pending Queue) was full.
*   `envoy_cluster_outlier_detection_ejections_active`: `1`
    *   **Meaning:** One pod is currently "Ejected" (kicked out) from the pool because it returned too many 5xx errors.

---

## 3. Downstream "Listener" Metrics
These tell you about the **Clients talking to YOU**.

| Metric | Example Value | Meaning |
| :--- | :--- | :--- |
| `envoy_listener_downstream_cx_active` | `10` | There are 10 active clients (like Fortio or other pods) connected to this sidecar. |
| `envoy_listener_downstream_cx_total` | `500` | Since the pod started, 500 different client connections have been made. |
| `envoy_listener_downstream_cx_destroy` | `490` | 490 connections have been closed (leaving 10 active). |

---

## 4. The Management Plane (xDS) Metrics
These tell you about the connection to **Istiod** (The Control Plane).

| Metric | Example Value | Meaning |
| :--- | :--- | :--- |
| `envoy_cluster_upstream_cx_http2_total{cluster_name="xds-grpc"}` | `1` | One persistent gRPC stream is open to Istiod to receive configuration updates. |
| `envoy_cluster_upstream_cx_rx_bytes_total{cluster_name="xds-grpc"}` | `11206408` | You have downloaded ~11MB of configuration data. |
| `envoy_cluster_update_failure{cluster_name="xds-grpc"}` | `0` | `0` is GOOD. If it's `>0`, the sidecar failed to apply the config sent by Istiod. |

> **Note on xds-grpc:**
> The `cluster_name="xds-grpc"` label refers to the **Control Plane connection**. It indicates the health and traffic of the "Management Lane" between your Sidecar and **Istiod**. It is where your proxy receives LDS, RDS, CDS, and EDS updates.

## 5. Service-Level Metrics (`istio_` prefix)
Istio takes the raw Envoy data and summarizes it into easier-to-read metrics starting with `istio_`. These are what Kiali and the standard Istio dashboards use.

### The "Label Power"
The main benefit of `istio_` metrics is that they are **Standardized**. Every service in your mesh provides the exact same labels (like `source_app`, `destination_service`, `response_code`).

| Metric | Meaning |
| :--- | :--- |
| `istio_requests_total` | A **COUNTER** incremented for every request. Used for throughput (RPS). |
| `istio_request_duration_milliseconds_bucket` | A **HISTOGRAM** for calculating Latency Percentiles (P50, P99). |
| `istio_request_messages_total` | Used for gRPC stream message counts. |
| `istio_tcp_sent_bytes_total` | For non-HTTP (L4) traffic, tracks data volume. |

---

## 6. Customizing Service-Level Metrics (Telemetry API CRD)
While you customize **Proxy** metrics (`envoy_`) via bootstrap config using `proxyStatsMatcher`, you customize **Service-Level** metrics (`istio_`) using the `Telemetry` resource. This allows you to add, update, or remove **Prometheus Labels** (Tags).


### A. Yaml Example:

```yaml
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: customize-metrics
  namespace: istio-system # Apply mesh-wide
spec:
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - match:
            metric: REQUEST_COUNT # This targets 'istio_requests_total'
          # In the world of Prometheus, a Tag is the same thing as a Label.
          # tagOverrides allows you to ADD, UPDATE, or REMOVE these labels.
          # The key (like destination_port) becomes the name of the label in your Prometheus output.
          tagOverrides:
            # 1. ADDING A LABEL: 'destination_port'
            # value: uses CEL to grab the actual port number from Envoy's metadata
            destination_port:
              value: "string(destination.port)"
            
            # 2. ADDING A LABEL: 'request_host'
            # value: grabs the 'Host' header from the HTTP request
            request_host:
              # The value: "request.host" part is a CEL (Common Expression Language) expression.
              # Internal to Envoy, request.host is a "variable" that points to the actual HTTP Host header sent by the client.
              # By putting this in a Telemetry resource, you are telling Istio: "Extract the Host header from every request and save it as a label so I can see it in my Prometheus dashboard."
              value: "request.host" 
            
            # 3. REMOVING A LABEL:
            # If you don't need 'response_code', you can drop it to save space
            response_code:
              operation: REMOVE
```

### B. Example: The "Before and After"
*   **Default:**         `istio_requests_total{reporter="destination", destination_app="httpbin", response_code="200"}`
*   **After Overrides:** `istio_requests_total{reporter="destination", destination_app="httpbin", destination_port="8080", request_host="httpbin.default.svc.cluster.local:8000"}`<br/>
(Note: `response_code` is gone if removed).

### C.1. Step-by-Step: How Overrides Work

The `Telemetry` resource uses:
* `Telemetry.spec.metrics[]Object.providers[]Object` list of metric providers e.g. `prometheus`.
* `Telemetry.spec.metrics[]Object.overrides[]Object` list to change how metrics are generated.
* `Telemetry.spec.metrics[]Object.overrides[]Object.match<Object>` is a block that determines **which** metric your changes apply to.
* `Telemetry.spec.metrics[]Object.overrides[]Object.tagOverrides<map[string]Object>` is a map of **Prometheus Label Names** to **CEL Expressions**.
Here is what those fields actually mean:


#### 1. Targeting (`metric` & `customMetric`)
The `match` block determines **which** metric your changes apply to.
*   **`Telemetry.spec.metrics[]Object.overrides[]Object.match<Object>.metric<Enum>`**: This is for the "Istio friendly name" metrics. It uses a specific set of keywords (Enums):
    *   `REQUEST_COUNT` → maps to `istio_requests_total`
    *   `REQUEST_DURATION` → maps to `istio_request_duration_milliseconds`
    *   `REQUEST_SIZE` → maps to `istio_request_bytes`
*   **`Telemetry.spec.metrics[]Object.overrides[]Object.match<Object>.customMetric<string>`**: This is a string field. If you have generated a non-standard metric (e.g., via a Wasm plugin or Lua script), you type its exact name here. 
    *   *Warning:* Istio doesn't validate this string. If you typo it, the override simply won't match anything.
*   **Pro Tip**: If you leave the `match` block empty, the overrides apply to **ALL** standard Istio metrics at once.

#### 2. Workload Mode (`Telemetry.spec.metrics[]Object.overrides[]Object.match<Object>.mode<Enum>`)
This tells Istio which "link" in the chain to modify. Since every request has two proxies involved (Source sidecar and Destination sidecar):
*   `SERVER`: Apply changes to **Inbound** metrics (the sidecar receiving the request).
*   `CLIENT`: Apply changes to **Outbound** metrics (the sidecar sending the request).
*   `CLIENT_AND_SERVER`: Apply to both (the default).

#### 3. Tag Overrides (`Telemetry.spec.metrics[]Object.overrides[]Object.tagOverrides<map[string]Object>`)
A **Tag** in Istio configuration becomes a **Label** in Prometheus. `tagOverrides` lets you manipulate these labels using **CEL (Common Expression Language)**.
*   **The Key**: The name of the label you want (e.g., `request_host`).
*   **The Value**: A **CEL (Common Expression Language)** expression. 
    *   Example: `"request.host"` extracts the Host header.
    *   Example: `"string(destination.port)"` extracts the port.
*   **The Operation**: You can also use `operation: REMOVE` to hide a label and save Prometheus storage space, `operation: UPSERT` to add or update a label.<br/>
E.g., the following snippet adds a `request_host` label and removes the `response_code` label:
```yaml
tagOverrides:
  request_host: # it becomes prometheus label 'request_host="<value: request.host>"'
    value: "request.host"
  response_code:
    operation: REMOVE
```


### D. The Hierarchy: How Istio Handles Multiple Resources
You can have `Telemetry` resources at three levels:
1.  **Mesh-wide (Global)**: Created in the `istio-system` namespace.
2.  **Namespace**: Created in a specific namespace (e.g., `default`).
3.  **Workload Selector**: Targeting specific pods using labels.

#### 1. Merge vs. Overwrite (The "ABCD" Rule)
*   **Merge (Additive)**: If the Global resource changes `REQUEST_COUNT` and the Namespace resource changes `REQUEST_DURATION`, the workload sees **BOTH**.
*   **Overwrite (Replacement)**: If BOTH resources try to change the **same** metric (e.g., `REQUEST_COUNT`), the more specific one **completely replaces** the other.
    *   *Example:* If Global has `tags: [A, B]` and Namespace has `tags: [C, D]` for the same metric, the result is **only [C, D]**. It does NOT merge them into `ABCD`.

#### 2. What is a Workload Selector?
A workload selector allows you to apply a `Telemetry` policy to only a specific set of pods within a namespace (like a "Laser Pointer"). It uses standard Kubernetes labels.

```yaml
spec:
  # This targets ONLY pods with the label 'app: my-app'
  selector:
    matchLabels:
      app: my-app
```

### E. Rules of the Telemetry API CRD
1.  **Scope**: The hierarchy is **Mesh-wide > Namespace > Workload**.
2.  **Uniqueness**: Only **one** Telemetry resource can apply to a specific workload at a time. If two resources in the same namespace target the same workload, Istio's behavior is undefined (don't do it!).
3.  **The Logic Engine**: The `value:` field uses **CEL (Common Expression Language)**.
    *   *Deep Dive:* See the **[Appendix: CEL vs. Starlark](Appendix-CEL-Starlark.md)** for a comparison of the languages.

## 7. Comparison & Debugging
One of the best ways to use these metrics is to **compare** behavior across the mesh.

1.  **"My app is slow"**: Check `envoy_cluster_upstream_rq_time` (Upstream latency) vs total time. This tells you if the bottleneck is your code or the network.
2.  **"My config isn't working"**: Check `envoy_cluster_update_success{cluster_name="xds-grpc"}`. If it hasn't increased since you applied your YAML, the sidecar hasn't received the update.
3.  **"I'm getting 503 errors"**: Check `envoy_cluster_upstream_rq_pending_overflow`. If it is increasing, your **Circuit Breaker** is too tight.
