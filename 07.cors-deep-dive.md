# Chapter 7 — CORS Deep Dive (Cross-Origin Resource Sharing)

In modern web development, your frontend (e.g., `https://shop.com`) and your backend (e.g., `https://api.shop.com`) are often on different domains or subdomains. By default, browsers block this for security. This chapter explains exactly how to handle this safely.

---

## 1. The Core Concept: Origin
An **Origin** is the combination of three things:
1. **Scheme** (http vs https)
2. **Host** (domain.com vs api.domain.com)
3. **Port** (80 vs 8080)

If **any** of these differ, the request is considered **Cross-Origin**.

### The Same-Origin Policy (SOP)
The browser's "Same-Origin Policy" is a built-in security wall. It says: *"JavaScript on my page can send a request to another domain, but I (the browser) will not allow the JavaScript to read the response unless that domain explicitly gives permission."*

---

## 2. What are the "Resources"?
When we talk about "Resource Sharing", what exactly is being shared?

- **JSON/XML Data:** The actual body of your API response (e.g., product lists, order details).
- **HTTP Headers:** Custom headers like `x-api-key` or `Authorization`.
- **Cookies & Credentials:** Session tokens or login cookies.

---

## 3. The "Credentials" and Cookie Ownership
This is the most misunderstood part of CORS.

### `credentials: 'include'`
When you write `fetch(url, { credentials: 'include' })` in your frontend code:
- It **DOES NOT** send cookies belonging to the site the user is currently visiting (`frontend.com`).
- It **DOES** send cookies belonging to the **Target Domain** (`api.backend.com`).

### The Real-World Flow
1. **Monday:** The user logs into `api.backend.com` directly. The browser saves a cookie: `session_id=123; Domain=api.backend.com`.
2. **Tuesday:** The user visits `https://frontend.com`.
3. **The Fetch:** The JavaScript on `frontend.com` calls your API with `credentials: 'include'`.
4. **The browser's job:** The browser sees the destination is `api.backend.com`. It looks in its "cookie jar", finds `session_id=123`, and attaches it to the request invisibly.

> **Ownership belongs to the receiver.** One domain can never "see" or "steal" cookies from another domain. The browser simply delivers the cookies to their rightful owner.

---

## 4. The CORS Handshake (Simple vs Preflight)

### 4.1 Simple Requests
(GET or POST with standard `text/plain` content types).
1. Browser sends the request immediately with an `Origin: https://frontend.com` header.
2. Server responds with the data and the header `Access-Control-Allow-Origin: https://frontend.com`.
3. Browser lets the JavaScript read the data.

### 4.2 Preflight (The OPTIONS Request)
For "dangerous" requests (DELETE, PUT, or any request with a JSON body or custom headers):
1. **The Question:** Browser sends an `OPTIONS` request first. *"Hey API, am I allowed to send a DELETE request with an Authorization header from frontend.com?"*
2. **The Answer:** Server responds (without data) with allowed methods and origins.
3. **The Action:** If the server says "Yes", the browser then sends the **actual** DELETE request.

---

## 5. Security Risks & Misconfigurations

### Risk 1: The Wildcard (`*`)
Setting `Access-Control-Allow-Origin: *` allows **any** website in the world (including malicious ones) to make a hidden request to your API and read the data if the user is visiting that site.
- **Rule:** Never use `*` if your API handles private user data.

### Risk 2: Echoing back the Origin
Some developers write code that simply reads the `Origin` header from the request and prints it back in the response header `Access-Control-Allow-Origin`.
- **Risk:** This is the same as using `*`. It makes the security wall useless.
- **Fix:** Use a **Whitelist** (an array of allowed domains).

### Risk 3: Missing `Vary: Origin`
If your server is behind a CDN or Cache, the cache might store the CORS response for `frontend.com` and accidentally serve it to `malicious.com`.
- **Fix:** Always send `Vary: Origin` to tell caches that the CORS headers depend on the sender.

## 6. Practical Implementation

### 6.1 Backend: Node.js + Fastify
Fastify uses a plugin that handles all the `OPTIONS` logic and header setting for you.

```javascript
// server.js
const fastify = require('fastify')();

// Define your trusted list
const whitelist = ['https://frontend.com', 'https://admin-panel.com'];

fastify.register(require('@fastify/cors'), {
  // Logic to check origin against whitelist
  origin: (origin, cb) => {
    if (!origin || whitelist.includes(origin)) {
      cb(null, true); // Allow
      return;
    }
    cb(new Error("Not allowed by CORS"), false); // Block
  },
  
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  
  // CRITICAL: Must be true if you use credentials: 'include' in frontend.
  // This sends 'Access-Control-Allow-Credentials: true' to the browser.
  credentials: true, 
  
  // How long the browser should cache the 'OPTIONS' result (so it doesn't ask every 2 seconds)
  maxAge: 86400 // 24 hours
});

/* 
  UNDER THE HOOD:
  When a request from 'https://frontend.com' hits the server, 
  the @fastify/cors library automatically adds these headers to YOUR response:
  
  1. Access-Control-Allow-Origin: https://frontend.com
  2. Access-Control-Allow-Credentials: true
  3. Access-Control-Expose-Headers: <any headers you defined>
  4. Access-Control-Max-Age: 86400 (The "OPTIONS" cache time)
  5. Access-Control-Allow-Methods: GET, POST, PUT, DELETE
  6. Access-Control-Allow-Headers: Content-Type, Authorization
  7. Vary: Origin (Crucial for CDN caching security)
*/

fastify.get('/orders', async (request, reply) => {
  // You DO NOT need to manually set headers here.
  // Because 'https://frontend.com' is in your whitelist, the @fastify/cors plugin
  // will automatically add 'Access-Control-Allow-Origin: https://frontend.com' 
  // and 'Vary: Origin' to this response before it leaves the server.
  return { orders: [{ id: 1, total: 99.99 }] };
});

fastify.listen({ port: 3000 });
```

### 6.2 Frontend: Fetch API
```javascript
// app.js on https://frontend.com

async function getOrders() {
  try {
    const response = await fetch('https://api.backend.com/orders', {
      method: 'GET',
      
      // 'include' -> sends api.backend.com's cookies
      // 'same-origin' -> only sends cookies if URL is on the same domain (default)
      // 'omit' -> never sends cookies
      credentials: 'include',
      
      headers: {
        'Content-Type': 'application/json',
        'Authorization': 'Bearer my-token'
      }
    });

    if (!response.ok) throw new Error('Network response was not ok');
    
    const data = await response.json();
    console.log('Orders:', data);
    
  } catch (error) {
    // If CORS fails, you end up here.
    console.error('CORS Error or Network Failure:', error);
  }
}
```

## 7. How Istio fits in
When using Istio, you can move the CORS logic **out of your code** and into the Envoy proxy. This is great for an eCommerce platform because it ensures consistent security across all your microservices (Product, Order, Payment).

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: order-service-vs
spec:
  hosts:
  - api.backend.com
  http:
  - route:
    - destination:
        host: order-service.default.svc.cluster.local
    corsPolicy:
      allowOrigins:
      - exact: "https://frontend.com"
      allowMethods:
      - GET
      - POST
      allowHeaders:
      - Authorization
      - Content-Type
      allowCredentials: true
      maxAge: "24h"
```

## 8. Clarifying the `credentials: true` flag

When you set `credentials: true` in your backend (Fastify or Istio), you are telling the browser: **"I explicitly permit this request to carry sensitive identity information."**

### What does the browser do when it sees this?

If the browser receives a response that **lacks** `Access-Control-Allow-Credentials: true` but the frontend used `credentials: 'include'`:
1. The browser **immediately blocks** the response.
2. It throws a console error: *"The 'Access-Control-Allow-Credentials' header in the response is '' which must be 'true' when the request's credentials mode is 'include'."*
3. The JavaScript `fetch()` call fails.

### Does it "share" credentials?
No. It does not allow `frontend.com` to see the cookies of `api.backend.com`. It only allows the **delivery** of those cookies to the server.

### What about Authorization Headers?
The `Authorization` header (`Bearer <token>`) is also considered a "credential". 
- In a **same-origin** request, you can send it freely. 
- In a **cross-origin** request, if you include an `Authorization` header, the browser considers the request "credentialed". 
- Therefore, even if you are **not** using cookies, but you are using an `Authorization` header, you often need `credentials: true` on the backend and `credentials: 'include'` on the frontend for most browsers to handle the cross-origin handshake correctly for non-simple requests.

---

## Summary of the "Credentials" Handshake

| Frontend Setting | Backend Setting | Result |
| :--- | :--- | :--- |
| `credentials: 'omit'` | `credentials: false` | No cookies sent. Request works. |
| `credentials: 'include'` | `credentials: false` | Cookies sent → **Browser BLOCKS response**. |
| `credentials: 'include'` | `credentials: true` | Cookies sent → **Success**. |
| `credentials: 'include'` | `origin: '*'` | **Browser BLOCKS response** (Security risk: Credentialed requests cannot use wildcards). |

## 9. CORS Headers vs. Cookie Attributes (Don't mix them up!)

You mentioned **SameSite**, **Secure**, and **HttpOnly**. These are **NOT** CORS headers; they are **Cookie Attributes**. However, they are the "other half" of the puzzle when you want cross-origin requests to work with cookies.

### 9.1 The `Access-Control-Max-Age` Header
When you set `maxAge: 86400` in Fastify or Istio, the server sends the header:
`Access-Control-Max-Age: 86400`
- **What it does:** It tells the browser: "You don't need to send an `OPTIONS` (preflight) request for every single fetch. You can remember that `frontend.com` is allowed to call `api.backend.com` for the next 24 hours."
- **Benefit:** It makes your app much faster because it removes the extra round-trip "preflight" delay.

### 9.2 The Cookie Security Attributes
For your eCommerce project, when your backend sets a cookie (`Set-Cookie`), you must use these attributes for it to work with CORS:

| Attribute | What it does | Required for CORS? |
| :--- | :--- | :--- |
| **`HttpOnly`** | Prevents JavaScript from reading the cookie. (Protects against XSS). | **Recommended** for security. |
| **`Secure`** | Cookie is only sent over HTTPS. | **REQUIRED** if `SameSite=None`. |
| **`SameSite=None`** | **This is the big one.** It tells the browser: "This cookie is allowed to be sent in cross-origin requests." | **REQUIRED** for `credentials: 'include'`. |
| **`SameSite=Lax`** | Default in modern browsers. Only sends cookies for same-site requests or top-level navigation. | Will **BLOCK** your cross-origin fetch cookies. |

### The "Perfect Pair" for your Shop
For a cross-origin login to work:
1. **The Cookie:** Must be sent from the backend as `Set-Cookie: session=xyz; SameSite=None; Secure; HttpOnly`.
2. **The CORS:** Backend must send `Access-Control-Allow-Credentials: true`.
3. **The Fetch:** Frontend must use `credentials: 'include'`.

If any one of these is missing, the user will be logged out the moment they try to use the frontend!

## 10. Advanced: Same-Site vs. Same-Origin (Subdomains)

Your intuition is excellent. There is a difference between **Origin** and **Site**.

### 10.1 The Difference
- **Origin:** (Scheme + Domain + Port). `frontend.store.com` and `backend.store.com` are **different origins**. You **STILL NEED CORS** headers.
- **Site:** (The TLD + 1, e.g., `store.com`). `frontend.store.com` and `backend.store.com` are **the same site**.

### 10.2 Does `Lax` work for subdomains?
**Yes.** Because they share the same registrable domain (`store.com`), they are considered **Same-Site**. 

- In this specific case, `SameSite=Lax` **will** work and the browser will send the cookies.
- You still need `credentials: 'include'` in your `fetch()` because the browser sees it as a cross-origin request.
- You still need `Access-Control-Allow-Credentials: true` and `Access-Control-Allow-Origin: https://frontend.store.com` on the backend.

### 10.3 Why would you still use `SameSite=None`?
Many professional e-commerce platforms still use `SameSite=None; Secure` because:
1. **Flexibility:** If you later decide to move your frontend to `mystore-ui.vercel.app` or a different TLD, `Lax` will immediately break.
2. **Public Suffixes:** If you use a service like `github.io` (e.g., `user1.github.io` and `user2.github.io`), the browser treats them as **Cross-Site** because `github.io` is on the "Public Suffix List". In that scenario, `Lax` would fail.

**Summary for your Subdomain Setup:**
- **Origin:** Different (Need CORS).
- **Site:** Same (Lax works).
- **Recommendation:** Use `Lax` if you are 100% sure you will always stay on subdomains of `store.com`. Use `None` for maximum compatibility across different environments.
