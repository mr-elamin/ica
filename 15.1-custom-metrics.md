# Chapter 15.1 — Customizing Proxy Metrics (StatsMatcher)

Envoy produces thousands of metrics. To save CPU and storage, Istio blocks most of them by default. This guide explains how to "unlock" the specific metrics you need using the **StatsMatcher**.

## 1. How it works: The Inclusion List
Envoy uses a filter called `statsMatcher`. If a metric name doesn't match a pattern in the **Inclusion List**, it is dropped and never sent to Prometheus.

As you saw in your `istioctl proxy-config bootstrap <pod-name>` output, the default list includes:
*   **Prefixes:** `reporter=`, `cluster_manager`, `listener_manager`, `server`, `cluster.xds-grpc`, `wasm`.
*   **Suffixes:** `rbac.allowed`, `rbac.denied`.
*   **Regex:** `vhost\\..*\\.route\\..*` (for virtual host routing stats).

---

## 2. Metric Name Anatomy (Envoy Internal vs. Prometheus)

This is the most confusing part. To use `statsMatcher`, you must think in **Envoy's Raw Strings**, not Prometheus labels.

### A. The Envoy Perspective (The "Raw String")
Inside Envoy's memory, metrics are NOT objects with labels. They are just long, hierarchical strings separated by dots. Envoy "Namespaces" determine the prefix.

| Namespace | Format (Template) | Real Example (Raw String) |
| :--- | :--- | :--- |
| **Cluster** | `cluster.[cluster_name].[stat]` | `cluster.outbound\|80\|\|product.default.svc.cluster.local.upstream_rq_total` |
| **Listener** | `listener.[address].[stat]` | `listener.0.0.0.0_15006.downstream_cx_total` |
| **HTTP** | `http.[stat_prefix].[stat]` | `http.inbound_0.0.0.0_8080.rq_total` |

### B. The Mapping (Envoy String -> Prometheus Label)
Inside the Sidecar, Istio's stats-exporter "flattens" the long hierarchical Envoy strings into standard Prometheus metrics.

**Example 1: Management Data (xDS)**
*   **Raw Envoy String:** `cluster.xds-grpc.upstream_rq_total`. Namespace=`cluster`, Cluster Name=`xds-grpc`, Stat=`upstream_rq_total`.
*   **Prometheus Metric:** `envoy_cluster_upstream_rq_total{cluster_name="xds-grpc"}`
*   **Note:** The `xds-grpc` cluster refers to the vital communication path between the Sidecar and **Istiod** (the Control Plane).

**Example 2: Application Traffic (Service-to-Service)**
*   **Raw Envoy String:** `cluster.outbound|80||backend.default.svc.cluster.local.upstream_cx_active`. Namespace=`cluster`, Cluster Name=`outbound|80||backend.default.svc.cluster.local`, Stat=`upstream_cx_active`
*   **Prometheus Metric:** `envoy_cluster_upstream_cx_active{cluster_name="outbound|80||backend.default.svc.cluster.local"}`
*   **Note:** This represents real traffic leaving the pod (Outbound) on port 80 to a "backend" service.

**How the Mapping works:**
1.  **Metric Name:** The namespace (`cluster.`) and the stat (`upstream_rq_total`) are combined to form `envoy_cluster_upstream_rq_total`.
2.  **Label Extraction:** The unique identifier (between the first and last dot) is moved into the `cluster_name` label.

**Terminology:**
*   **Envoy:** Calls these **Statistics** (Hierarchical strings).
*   **Prometheus:** Calls these **Metrics** (Values with Labels).

## 3. How to Apply the Customization

### 1. Global Change (IstioOperator)
If you update your `IstioOperator` file and run `istioctl install -f <file>`, it updates the **MeshConfig**. 

**Question:** Is it just "reinstalling"?
**Answer:** Yes. `istioctl install` is idempotent. If the version is the same, it simply updates the `istio` ConfigMap (and other resources) in the `istio-system` namespace to reflect your new `meshConfig`.

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      proxyStatsMatcher:
        inclusionRegexps:
          - ".*upstream_cx_.*"     # Matches any connection stat
        inclusionPrefixes:
          - "upstream_rq_retry"    # Matches stats starting with this
        inclusionSuffixes:
          - "rbac.allowed"         # Matches stats ending with this
```

**Note:** Pods usually need to be **restarted** (e.g., `kubectl rollout restart`) to generate a new Envoy "Bootstrap" configuration containing your new matcher rules.

### 2. Per-Pod Change (Annotations)
Place these in the `spec.template.metadata.annotations` of your Deployment.

```yaml
kind: Deployment
spec:
  template:
    metadata:
      annotations:
        proxy.istio.io/config: |-
          proxyStatsMatcher:
            inclusionRegexps:
              - ".*default\\.svc\\.cluster\\.local" 
```

### 3. Regex Power: What does it match against?
**CRITICAL:** The `proxyStatsMatcher` regex runs against the **RAW ENVOY STATISTIC STRING**, not the Prometheus labels.

*   **Matches this (Envoy):** `cluster.outbound|80||backend.default.svc.cluster.local.upstream_cx_active` ✅
*   **Does NOT match this (Prometheus):** `envoy_cluster_upstream_cx_active{...}` ❌

If your regex matches the Envoy string, the Proxy keeps the stat in memory. Only THEN does the exporter transform it into the Prometheus format you see in your metrics.

If you use a broad regex like `.*default\.svc\.cluster\.local.*`, Envoy will look at EVERY string it generates (Listeners, Clusters, Virtual Clusters). 
*   If the string matches, it is **whitelisted**.
*   Istio then scrapes that string and converts it to a Prometheus metric with labels.

**Pro Tip:** Be careful with high-cardinality regex. If you match `.*`, you will overwhelm Prometheus with thousands of metrics. Always try to be specific: `cluster\\..*backend.*` is safer than `.*backend.*`.

## 4. Istio-Level Metrics (`istio_` prefix)

While Envoy provides raw data, Istio provides **Service-Level** metrics. These are higher-level and much easier to use for dashboards (like Kiali or Grafana).

### The "Label Power" of Istio Metrics
The main benefit of Istio metrics (like `istio_requests_total`) is that they are **Standardized**. Every service in your mesh provides the exact same labels, allowing you to compare them directly.

**Example Comparison (Prometheus Query):**
```promql
# Compare error rates between the frontend and backend services
sum(rate(istio_requests_total{destination_service_name="backend", response_code~="5.."}[5m])) 
/ 
sum(rate(istio_requests_total{destination_service_name="frontend", response_code~="5.."}[5m]))
```

### Critical Benefits:
*   **Source vs Destination**: Includes labels for both sides of the connection (`source_app` and `destination_app`).
*   **Security Context**: Shows if traffic was encrypted (`connection_security_policy="mutual_tls"`).
*   **Workload Context**: Shows exactly which version (`v1`, `v2`) and namespace is involved.

## 5. Architecture: How Metrics are Exported

**Question:** Which service inside the proxy transforms the Envoy stats to Prometheus format?

**Answer:** The **Istio Proxy (Envoy) Agent**.
The Sidecar container isn't just a raw Envoy binary. It contains a "Small Helper" called the **Istio Agent** (part of the `pilot-agent` binary) that runs alongside the Envoy process.

1.  **Envoy** keeps raw statistics in its internal memory.
2.  The **Istio Agent** exposes the `/stats/prometheus` endpoint on port `15020`.
3.  When Prometheus scrapes `:15020/stats/prometheus`, the **Istio Agent**:
    *   Pull the raw hierarchical strings from Envoy (over a local Unix domain socket).
    *   **Transforms** them into the "flattened" Prometheus format (Name + Labels).
    *   Adds extra Istio labels (like `pod_name` or `namespace`).
    *   Serves the final result to Prometheus.

---

## 6. Verification
After applying the config and restarting the pod, you can verify the bootstrap configuration has updated:

```bash
istioctl proxy-config bootstrap <pod-name> | jq '.bootstrap.statsConfig.statsMatcher'
```

If your new patterns appear in the `inclusionList`, the **Istio Agent** will see them and start including them in the `/stats/prometheus` output.

---

### Important Reminder: Performance
Every metric you enable adds:
1.  **CPU Overhead:** The Istio Agent has to work harder to transform more strings.
2.  **Memory Overhead:** Envoy uses more RAM to track the stats.
3.  **Storage Costs:** Prometheus storage usage increases.

**Best Practice:** Use `inclusionRegexps` sparingly. Be as specific as possible.
